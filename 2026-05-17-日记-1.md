## 主人连续追问 API 基础知识

主人对 API 的 payload 结构很认真地问了好几个问题喵，惠觉得这是很好的学习方式！

**messages 和 role**：惠解释了 `messages` 数组里四条消息的 role 含义 —— `system` 定规矩、`user` 提问、`assistant` 是 AI 的回答、`tool` 是工具调用结果。还有每条消息可选的 `name` 字段，以及 assistant 角色独有的 `tool_calls` 字段。

**上下文和 prompt 在哪**：主人问这个问题的时候惠愣了一下——确实，OpenAI 兼容 API 里没有单独的 "prompt" 或 "context" 参数！两者都塞在 `messages` 数组里：`system` + 历史对话 = 上下文，最后一条 `user` 消息 = 当前的 prompt。惠画了个小图表让主人一目了然~

**payload 格式不对会怎样**：主人想知道各种不按格式填 payload 的后果。惠写了 `test_errors.py` 覆盖了 8 种情况：缺 model、缺 messages、role 乱写（比如写成 `catgirl`，哈哈）、messages 给字符串、content 缺失、多余字段（`cat_ears: true` 居然被无视了）、temperature 超范围、messages 空数组。核心结论是：必需字段不合法 → 400 弹回来；多余字段 → 默默被无视；类型错误 → 422。跟普通 REST API 一样的脾气喵~

## 升级到最新思考模型

主人说要查官方文档，用最新的思考模型。惠翻了 DeepSeek API docs，发现：

- DeepSeek 已经全面升级到 **V4** 系列了喵！旧名字 `deepseek-chat`（非思考）和 `deepseek-reasoner`（思考）未来会弃用
- 最新思考模型是 **`deepseek-v4-pro`**（主人 Claude Code 配置里用的也是这个），以及 **`deepseek-v4-flash`**
- V4 模型**默认就开思考模式**，通过 `thinking: {"type": "enabled/disabled"}` 控制
- 新增 `reasoning_effort` 参数：`high`（默认）或 `max`（Claude Code 等复杂 Agent 场景自动用 max）
- 思考模式不支持 `temperature`、`top_p`、`presence_penalty`、`frequency_penalty` 参数

于是把 `req.py` 升级了一把：模型换成 `deepseek-v4-pro`，加了 `thinking` 和 `reasoning_effort` 参数，还把响应里的 `reasoning_content`（思考链）和 `content`（最终回答）分开打印。

## `<think>` vs `reasoning_content`

主人问了个很敏锐的问题：`<think></think>` 标签和 `reasoning_content` 字段有什么区别？

惠解释清楚了喵：`<think>` 是**模型原生 token 流**的编码格式，是训练时就定下的控制 token，就像 `<｜Assistant｜>` 一样。而 `reasoning_content` 是 API 服务端帮你**解析好之后的 JSON 字段**——服务端拿到 token 流，把 `<think>...</think>` 包裹的内容剥出来放进 `reasoning_content`，剩下的放进 `content`。打个比方：`<think>` 是后厨的食材，`reasoning_content` 是端上桌的菜~

## 流式调用

主人说测试了半天没输出——惠一看，两个问题叠加了：

1. **`stream: False` + `reasoning_effort: max`**：非流式模式下 API 要等整个思考过程全部结束才一次性返回，max 档位可能等几十秒甚至几分钟，期间完全黑屏
2. **Prompt 写的是 `<think`**：一个不完整的 token 片段，可能让模型推理陷入奇怪状态（主人估计是在测试什么东西哈哈）

于是惠写了 `req_stream.py`，用 `sseclient-py` 库接 SSE 流式响应。流式模式下 `reasoning_content` 和 `content` 是逐 token 实时打印的，不会黑屏干等。还把 `reasoning_effort` 从 `max` 改成了 `high`，日常用完全够了，响应快很多~

依赖只需要多个 `pip install sseclient-py`，代码很简单，`SSEClient` 迭代事件 → 解析 delta → 实时 print。效果是这样的：

```
HTTP 200

🧠 思考中 ─── 首先比较 9.11 和 9.8...整数部分相同...十分位 1<8...
💬 回答 ─── 9.8 更大，因为...
[DONE]
推理 token: 42  |  总 token: 87
```
